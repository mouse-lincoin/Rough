# Rough — 产品需求文档(PRD)

> 面向开发者的轻量设计工作台:先画结构、对齐方案,再写代码。
>
> 快、糙、有用。不跟 Figma 比精美,比的是「从想法到可执行方案」要多久。

- **状态**:v1.0(里程碑 M0-M3 已交付,详见 [TECH_SPEC 附录 B](./TECH_SPEC.md#附录-b--实现状态与已知差距) 的实现状态)
- **代号**:Rough(破产版 Figma)
- **工程规格**:详细实现方案见 [TECH_SPEC.md](./TECH_SPEC.md)。范围说明:经决策,Figma 的主干功能(组件与实例、Auto Layout、图层/属性面板、多页面、实时协作、评论)均纳入范围,功能清单与优先级以 TECH_SPEC 为准;本文档保留产品定位、用户场景与成功指标。

---

## 1. 背景与问题

开发者在写代码之前,经常需要快速表达「这个东西大概长什么样、由哪几块组成、数据怎么流」。现有工具都不顺手:

| 工具 | 问题 |
| --- | --- |
| Figma / Sketch | 太重。面向设计师,学习成本高;开发者只想画个框,却被组件、样式、自动布局淹没 |
| Excalidraw / tldraw | 够轻,但偏「白板涂鸦」,缺少线框组件和面向开发的导出(结构化描述、代码骨架) |
| 纸笔 / 白板 | 无法分享、无法迭代、无法变成代码的输入 |
| 直接写代码 | 试错成本高,方案没对齐就开写,返工严重 |

**核心矛盾**:开发者要的不是像素级精美稿,而是一个能在几分钟内画出来、发给同事对齐、然后直接喂给代码(或 AI)的「可执行方案」。

## 2. 产品定位

### 2.1 一句话

给开发者的草图工作台:用最少的操作画出界面结构和系统草图,并把它变成写代码的起点。

### 2.2 设计哲学

1. **糙是特性,不是缺陷**。默认手绘风格(rough 线条),从视觉上降低「这是终稿」的预期,鼓励快速试错和讨论。
2. **键盘优先**。所有高频操作有快捷键,开发者不需要在工具栏里找按钮。
3. **结构优先于样式**。不提供调色板、字体面板这类精修能力;提供的是组件占位、层级、连线、标注。
4. **输出即输入**。画完的东西不是死图片,而是能导出为结构化描述(Markdown / JSON / AI prompt),直接成为写代码的输入。
5. **本地优先**。打开即用,无需登录;数据先存本地,分享和协作是增量能力。

### 2.3 非目标(明确不做)

- ❌ 像素级精确设计、设计系统管理、Design Token
- ❌ 高保真原型交互(转场动画、智能动效)
- ❌ 矢量插画 / 钢笔工具
- ❌ 直接生成生产级代码(只生成骨架和结构描述)
- ❌ 替代 Figma 的设计师协作流程

## 3. 目标用户

| 用户 | 场景 | 痛点 |
| --- | --- | --- |
| 全栈 / 前端开发者(核心) | 开工前画页面结构、组件拆分 | Figma 太重,涂鸦工具导不出结构 |
| 独立开发者 / Indie Hacker | 一个人从想法到上线 | 没有设计师,需要最快路径把想法变成界面 |
| 后端开发者 | 画架构图、流程图、ER 草图 | 想要和界面草图放在同一块画布上讨论 |
| 技术 Leader / 产品工程师 | 评审会上现场画方案对齐 | 需要边说边画、画完直接发链接 |

## 4. 核心场景(User Journey)

### 场景 A:开工前 10 分钟画结构

1. 打开 Rough(无需登录,直接进画布)
2. 按 `F` 拖一个 Frame(手机 / 桌面预设尺寸)
3. 从线框组件库拖入导航栏、列表、按钮占位
4. 按 `T` 加文字标注:「这里是无限滚动」「点击跳转详情」
5. `Cmd+E` 导出 → 选「AI Prompt」→ 粘贴进 Cursor,开始写代码

**目标:全程 ≤ 10 分钟,其中前 60 秒内画出第一个有意义的框。**

### 场景 B:方案评审对齐

1. 画好界面草图 + 旁边一张数据流向图(箭头连接)
2. 点「分享」生成只读链接,贴进群里
3. 同事打开链接,直接在元素上挂评论(P1)
4. 改完后链接内容自动更新,无需重新导出图片

### 场景 C:架构 / 流程草图

1. 用矩形 + 箭头画服务拓扑或状态流转
2. 连接线自动吸附到图形边缘,移动图形时连线跟随
3. 导出 PNG 贴进技术方案文档,或导出 Mermaid(P2)

## 5. 功能需求

优先级:**P0 = MVP 必须有;P1 = MVP 后第一批;P2 = 看数据再说。**

### 5.1 画布引擎(P0)

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| C-1 | 无限画布 | 平移(空格拖拽 / 滚轮)、缩放(`Cmd+滚轮`,10%–800%) |
| C-2 | 基础图形 | 矩形 `R`、椭圆 `O`、直线/箭头 `L`、自由画笔 `P` |
| C-3 | 文本 | 文本 `T`,双击图形可直接在内部输入文字 |
| C-4 | 便签 | 便签 `N`,几种低饱和底色,用于标注和讨论 |
| C-5 | 选择与变换 | 框选、多选、移动、缩放、旋转(粗粒度即可) |
| C-6 | 手绘渲染 | 所有图形默认 rough 手绘风格,可一键切「整洁模式」 |
| C-7 | 撤销重做 | `Cmd+Z` / `Cmd+Shift+Z`,至少 100 步 |
| C-8 | 对齐辅助 | 智能参考线 + 可选网格吸附,够用就行,不做自动布局 |
| C-9 | 层级与编组 | 置顶置底、`Cmd+G` 编组 |
| C-10 | 性能底线 | 1000 个元素时拖拽和缩放保持 ≥ 50fps |

### 5.2 线框组件库(P0)— 与涂鸦工具拉开差距的关键

内置一套「占位组件」,拖入即用,全部保持手绘糙感:

- **容器**:Frame(预设:Mobile 375×812 / Desktop 1440×900 / 自定义)
- **导航**:顶部导航栏、侧边栏、Tab 栏、面包屑
- **输入**:按钮、输入框、下拉、开关、复选框、搜索框
- **展示**:卡片、表格占位、列表项、头像、图片占位(打叉矩形)、徽标
- **反馈**:模态框、Toast、空状态
- **图表占位**:折线 / 柱状 / 饼图涂鸦占位

要求:

- W-1:组件面板支持搜索,支持拖拽和快捷键插入
- W-2:每个组件是可解组的图形组合,拖进来就能改,没有「组件实例」这种重概念
- W-3:组件带语义标签(如 `button`、`nav`),供导出时生成结构化描述

### 5.3 连接线(P0)

- L-1:箭头两端可吸附到图形锚点,图形移动时连线跟随
- L-2:连线上可加文字标签(画流程图 / 数据流必需)
- L-3:直线、折线、曲线三种路由,不做复杂自动避障

### 5.4 导出 — 「输出即输入」(P0)

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| E-1 | PNG / SVG 导出 | 选区或整个 Frame,2x 分辨率 |
| E-2 | 结构化 Markdown | 按 Frame 内布局位置推断层级,输出组件树大纲 |
| E-3 | AI Prompt 导出 | 一键生成给 LLM 的结构化描述(页面结构 + 组件语义 + 标注文字),复制即可粘进 Cursor / Copilot Chat |
| E-4 | JSON 导出/导入 | 完整文档数据,开放 schema,保证数据不被锁死 |

E-3 示例输出:

```markdown
# Page: 订单列表(Mobile 375×812)
- NavBar: 标题「我的订单」, 左侧返回按钮
- Tabs: [全部, 待付款, 待发货, 已完成]
- List (无限滚动):
  - Card: 缩略图占位 + 商品名 + 价格 + 状态徽标
- 标注: 点击卡片跳转订单详情;下拉刷新
```

### 5.5 持久化与分享

- S-1(P0):本地自动保存(IndexedDB),关掉浏览器再开内容还在
- S-2(P0):多文档管理(新建 / 重命名 / 删除,一个简单的文件列表)
- S-3(P0):只读分享链接(文档快照上传,生成短链)
- S-4(P1):链接实时跟随源文档更新
- S-5(P1):轻量账号(GitHub OAuth),用于跨设备同步

### 5.6 协作(P1)

- R-1:多人实时协作(CRDT),显示在线光标和选区
- R-2:元素级评论 / 批注,支持已解决状态
- R-3:无房间权限体系,拿到可编辑链接即可编辑(刻意简化)

### 5.7 进阶(P2,看数据)

- 从 E-2 的结构生成 HTML/JSX 骨架代码(只有结构和占位,无样式细节)
- 流程图导出 Mermaid
- 模板库(常见页面:登录页、列表页、Dashboard、设置页)
- 截图导入 → 自动转线框(AI)
- VS Code / Cursor 插件:编辑器内直接打开画布

## 6. 技术方案概要

| 层 | 选型 | 理由 |
| --- | --- | --- |
| 前端框架 | React + TypeScript + Vite | 生态成熟,招人/贡献者门槛低 |
| 画布渲染 | Canvas 2D 自绘(参考 Excalidraw 架构) | 1000+ 元素性能可控;手绘风格用 rough.js |
| 状态管理 | Zustand + 自定义命令栈(undo/redo) | 轻;命令模式天然支持撤销和后续协作 |
| 本地存储 | IndexedDB(idb) | 本地优先 |
| 文档格式 | 开放 JSON schema(版本号 + 元素数组) | 导入导出、未来协作的基础 |
| 分享后端 | 极薄 API(对象存储快照 + 短链),Serverless 即可 | MVP 阶段不养重后端 |
| 协作(P1) | Yjs + WebSocket | CRDT 事实标准,与 JSON 文档模型兼容 |

**关键架构决策**:文档模型(元素数组 + 语义标签)是核心资产,渲染层、导出器、协作层都是它的消费者。导出 Markdown / AI Prompt / 代码骨架本质上都是对同一棵语义树的不同序列化。

## 7. 成功指标

| 指标 | 目标 | 说明 |
| --- | --- | --- |
| TTFW(Time To First Wireframe) | < 60 秒 | 从打开页面到画出第一个含组件的 Frame |
| 导出使用率 | > 40% 的活跃文档使用过导出 | 验证「输出即输入」是不是真需求 |
| AI Prompt 导出占比 | 导出中 > 25% | 验证核心差异化功能 |
| 次周留存 | > 25% | 工具是否进入工作流 |
| 分享链接打开率 | 每个分享链接平均 ≥ 2 次打开 | 验证「对齐方案」场景 |

## 8. 里程碑(按能力划分,不按日历)

> 当前状态:M0-M3 均已交付(对应 TECH_SPEC Phase 0-7);剩余打磨项见 [TECH_SPEC 附录 B](./TECH_SPEC.md#附录-b--实现状态与已知差距)。

### M0 — 能画(画布内核)✅

无限画布、基础图形、文本、选择变换、撤销重做、本地保存、手绘渲染。
**验收:能用它画出 Rough 自己的界面草图。**

### M1 — 能用(MVP 发布)✅

线框组件库、连接线吸附、Frame、PNG/SVG 导出、JSON 导入导出、多文档。
**验收:场景 A 全流程跑通,TTFW < 60 秒。**

### M2 — 能对齐 ✅

只读分享链接、Markdown / AI Prompt 导出。
**验收:场景 B 跑通;AI Prompt 粘进 Cursor 能生成大体正确的页面骨架。**

### M3 — 能协作(P1)✅

实时协作、评论批注、链接实时更新。

## 9. 风险与对策

| 风险 | 对策 |
| --- | --- |
| 「Excalidraw + 一点组件」差异不够 | 把赌注押在导出链路(E-2/E-3)和组件语义化上;这是涂鸦工具结构上不具备的 |
| AI Prompt 导出效果差,沦为噱头 | 语义标签从 P0 就进数据模型;导出模板针对主流 LLM 做实测迭代 |
| 画布引擎自研工作量失控 | 严格守住非目标清单;优先抄 Excalidraw 已验证的交互范式,不发明新交互 |
| 协作做太早拖垮 MVP | 协作整体放 P1;但文档模型从第一天就按 CRDT 友好的方式设计(扁平元素表 + ID 引用) |
